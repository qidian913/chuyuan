## 一、设计原则

### 1.1 核心原则

| 原则 | 说明 |
|------|------|
| **数据一生贯通** | 用户数据从孕期到老年期永不丢失，插件切换不丢失历史 |
| **底座克制插件丰富** | 底座只做跨阶段公共能力，功能差异化由插件实现 |
| **插件独立发布** | 各插件可独立开发、测试、发版、升级 |
| **隐私原生** | 公私空间隔离是架构层面的强制约束，非约定俗成 |

### 1.2 架构演进策略

```
Phase 1 (MVP)     Phase 2 (扩展)      Phase 3 (完善)
┌─────────────┐   ┌─────────────┐    ┌─────────────┐
```

### 2.3 数据模型关键设计

#### 2.3.1 贯穿一生的数据连接

```
用户A的一生数据流：

[孕期] ──────► [婴儿期] ──────► [幼儿期] ──────► ... ──────► [成年期] ──────► [老年期]
  │               │               │                           │               │
  │               │               │                           │               │
  ▼               ▼               ▼                           ▼               ▼
Life Record   Life Record     Life Record                   Life Record     Life Record
  │               │               │                           │               │
  └───────────────┴───────────────┴───────────────────────────┴───────────────┘
                                    │
                                    ▼
                            统一 Life ID = 一生数据贯通
```

**核心设计**：
- 每个用户有且仅有一个 `Life ID`，终身不变
- 所有阶段数据通过 `life_id` 关联
- 插件切换时，`life_id` 保持不变，数据完整保留

#### 2.3.2 公私空间隔离机制

```
┌──────────────────────────────────────────────────────────────┐
│                    数据访问控制矩阵                           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│   请求方 ──► 空间类型 ──► owner_life_id ──► 权限判定         │
│                                                              │
│   案例：                                                     │
│   用户B（儿子）查看 用户A（父亲）的私密日记                   │
│   → space_type=private                                      │
│   → owner_life_id=A                                        │
│   → 请求方 B ≠ owner (A)                                    │
│   → 权限=false → 拒绝访问                                   │
│                                                              │
│   用户B查看 家庭共享相册                                     │
│   → space_type=public                                       │
│   → family_id=相同家庭                                      │
│   → 权限=family → 允许访问                                  │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

**技术实现**：
```sql
-- 数据隔离查询示例
SELECT * FROM timeline 
WHERE space_type = 'public' 
  AND family_id IN (
    SELECT family_id FROM family_member WHERE life_id = :current_life_id
  )
OR (space_type = 'private' AND owner_life_id = :current_life_id)
```

```typescript
interface PluginManager {
  // 插件注册

interface PluginManifest {
  id: string;

  compatibleCoreVersion: string;
  
  stages: StageType[];

    init: () => Promise<void>;
    activate: () => Promise<void>;
    deactivate: () => Promise<void>;
    destroy: () => Promise<void>;
  };
```

```typescript
class InfantPlugin extends BasePlugin {
  async migrateToToddler() {
    // 1. 发送数据迁移请求
    await this.context.eventBus.publish({
      type: 'PLUGIN_MESSAGE',
      sourcePlugin: 'infant',
      targetPlugin: 'toddler',
      action: 'MIGRATE_DATA',
      payload: {
        lifeId: this.context.lifeId,
        dataTypes: ['feeding', 'sleep', 'health', 'milestone'],
        migrationDate: new Date()
      }
    });
    
    // 2. 请求底座切换阶段
    await this.context.stageManager.transitionTo('toddler');
  }
}

// 幼儿期插件接收数据
class ToddlerPlugin extends BasePlugin {
  async onEvent(event: AppEvent) {
    if (event.type === 'PLUGIN_MESSAGE' && event.action === 'MIGRATE_DATA') {
      await this.receiveMigration(event.payload);
    }
  }
  
  async receiveMigration(data: MigrationPayload) {
    // 处理来自婴儿期的数据
    await this.context.dataService.importData(data);
  }
}
```

```typescript
async retrieveMemory(lifeId: string, query: string): Promise<Memory> {

  // 2. 构建检索向量
  const embedding = await this.embeddingService.encode(query);

  const [personalResults, familyResults] = await Promise.all([
    this.vectorStore.search({
      collection: `memory_${lifeId}`,

      filter: { spaceType: 'public' }
    })
  ]);
  
  // 4. 融合排序
```

## 八、插件开发规范

### 8.1 插件注册流程

```typescript
// 1. 编写 manifest.json
// 2. 实现入口文件
import { Plugin, PluginContext } from '@shiguangshu/core';

export default class MyPlugin extends Plugin {
  static manifest = require('./manifest.json');
  
  async onInit(context: PluginContext) {
    // 注册页面
    context.pageRouter.register(this.constructor.manifest.pages);
    
    // 注册Widget
    context.widgetRegistry.register('my-widget', MyWidget);
    
    // 注册AI能力
    context.aiRegistry.register('my-ai', MyAIHandler);
  }
}

// 3. 打包发布
// shiguangshu-cli plugin build --plugin infant --output ./dist
```

### 8.2 插件安全规范

| 规范 | 说明 |
|------|------|
| 沙箱隔离 | 插件运行在独立沙箱，禁止访问系统API |
| 数据边界 | 插件只能操作自己创建的数据 |
| API限制 | 禁止调用未在manifest中声明的API |
| 代码审核 | 插件上架前需通过安全审核 |
| 版本签名 | 插件包需数字签名防篡改 |
